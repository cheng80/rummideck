# CURSOR.md

## 목적
Cursor IDE에서 코드 생성/수정 시 프로젝트 규칙을 일관되게 적용하기 위한 가이드.  
`AGENTS.md`와 **동일한 공통 규약**을 공유한다.

---

## 워크플로 (작업 방식)

### 1. 계획 먼저, 승인 후 코딩
- 코드를 작성하기 **전에** 접근 방식을 짧게 설명하고, 필요하면 승인을 기다린다.
- 요구 사항이 모호하면 **모든 구현 전에** 질문으로 범위를 맞춘다.

### 2. 큰 작업은 작게 분해
- **3개 이상의 파일**을 바꿀 때는 먼저 작업을 쪼개고, 단계별로 진행한다.
- 필요 시 단계마다 사용자 확인을 받는다.

### 3. 코드 작성 후 영향 분석
- 변경 후 **깨질 수 있는 지점**을 나열한다.
- 그에 맞는 **테스트를 제안**하거나 실행한다.

### 4. 버그 수정 시 테스트 우선
- 버그가 있으면 **재현 테스트를 먼저** 두고, 통과할 때까지 수정한다.

### 5. 교정 시 규칙 반영
- 같은 실수를 반복하지 않도록, 사용자가 **규칙 추가·수정을 요청하면** `CURSOR.md` / `AGENTS.md`에 반영한다.

## 기본 원칙
- **Docs-first** (아래 참조 순서 문서 우선)
- **작은 단위 변경**
- **실행 가능한 코드 유지**

## 구현 규칙
- Flutter + Flame  
- UI / Logic 분리  
- Seed 기반 RNG (`Random()` 금지)  
- RunContext 중심 상태  
- **UI 모듈화**: 반복되거나 복잡한 화면은 위젯/파일로 분리한다.  
- **View vs 로직**: 화면(`views`)은 UI·바인딩 위주, 세션/상태는 `vm`·`logic` 등 기존 구조를 따른다.  
- **한글 주석**: 핵심 흐름·비명확한 부분에 **짧은 한글 주석**을 둔다.  

### UI 코딩
- **Row / Column 동일 간격**: 가능하면 `spacing` 파라미터를 사용한다 (`SizedBox` 남발 지양).

### 실행 환경 (로컬)
- **우선**: iOS 시뮬레이터, **Debug** 모드로 검증.  
- macOS·웹은 보조.

### vm / 상태 네이밍 (참고)
- Riverpod **Notifier** 등 상태는 역할이 드러나게 이름 짓는다.  
- Git **Repository**와 혼동되도록 데이터 접근 클래스에 `Repository`만 붙이는 것은 피한다(프로젝트 관행에 맞출 것).

## 문서화 정책
- **게임 규칙·아키텍처**: 위 참조 순서 문서를 **깨지 않게** 유지·동기화한다.  
- **임의 신규 `.md`**: 사용자가 **문서로 남기라고 요청하지 않은 경우**, 요약·플랜만 대화로 제공하고 **새 마크다운 파일을 만들지 않는다.**  
- 예외: 이미 합의된 `docs/` 관행(예: [`docs/refactoring.md`](docs/refactoring.md) 및 `docs/archive/refactoring/` 아카이브)에 맞는 갱신은 허용.
